Category 1 — Software Design & Engineering

YAML Configuration Framework

Artifact: config.py · config.yaml · configuration variants

View Code ↗

Artifact Description. The original backtesting engine scattered 47 hardcoded parameters — initial capital, commission rates, stop-loss thresholds, moving-average windows, database connection strings — across three source files. Every experiment required a code change. This enhancement introduced a centralized YAML configuration framework: a ConfigManager class that loads any config file, provides dot-notation access to nested values, and substitutes environment variables for sensitive credentials.

Justification for Inclusion. This artifact demonstrates that I understand configuration as an architectural concern, not an afterthought. Separating code from configuration is a professional engineering practice with measurable benefits: reproducibility (save a YAML file, reproduce any result), parameter sweeping (test ten combinations without touching source code), and environment parity (dev/test/prod use the same code with different configs). The implementation follows Dependency Injection principles — the engine receives configuration as a parameter rather than reading globals — which also makes unit testing straightforward.

Reflection. Building this forced me to think about where parameters live, how validation happens, and how different environments coexist. I initially treated configuration as "move numbers to a file." I now treat it as a system with its own design trade-offs: compile-time vs. runtime validation, hierarchical vs. flat structure, versioned-with-code vs. externally managed. The environment variable substitution feature came directly from recognizing that database passwords must never be committed to source control.

# config.py — ConfigManager with dot-notation access
class ConfigManager:
    def get(self, key_path: str, default=None):
        """Access nested YAML values with dot notation.
        
        Args:
            key_path: Dotted path e.g. 'backtest.initial_cash'
            default:  Value returned if the key is absent
        """
        keys = key_path.split(".")
        value = self._config
        for key in keys:
            if not isinstance(value, dict) or key not in value:
                return default
            value = value[key]
        return self._substitute_env_vars(value)

Category 2 — Algorithms & Data Structures

Position Sizing Strategy Module

Artifact: BacktestingEngine/positionSizer.py · performanceMetrics.py

View Code ↗

Artifact Description. The original engine traded a single fixed share per signal — a placeholder, not a strategy. This enhancement replaced it with four professional position sizing algorithms implemented through a polymorphic class hierarchy: an abstract PositionSizer base class, four concrete implementations, and a factory function for runtime strategy selection.

Justification for Inclusion. Position sizing is where algorithms meet financial mathematics and engineering pragmatism. The Kelly Criterion implementation required genuine probability theory and is used by professional traders and hedge funds precisely because it is mathematically optimal. But implementing full Kelly also revealed a core engineering truth: mathematical optimality and practical optimality are different things. Full Kelly is theoretically best in expectation but produces aggressive, high-variance positions. Professional implementations use fractional Kelly (typically ¼ Kelly) to reduce variance while preserving most of the return advantage. This trade-off is not obvious from the math alone — it requires domain knowledge.

Reflection. The polymorphic architecture (Strategy + Factory patterns) was a deliberate choice that pays dividends: adding a fifth sizing strategy requires only a new class and a one-line factory addition — no changes to the engine, tests, or configuration system. The volatility-adjusted sizer was the most interesting to design because it required a stateful running volatility calculation with inverse scaling logic: the engine must maintain history between trades, not just compute a per-trade value.

# Strategy pattern — abstract base + 4 concrete implementations
class PositionSizer(ABC):
    @abstractmethod
    def calculate_position_size(self, account_value, current_price, **kwargs):
        pass

class KellyCriterionPositionSizer(PositionSizer):
    """Mathematically optimal sizing via Kelly Criterion.
    
    Uses fractional Kelly (default ¼) for practical variance management.
    Formula: f* = (p×b - q) / b  where p = win_rate, b = profit/loss ratio
    """
    def calculate_position_size(self, account_value, current_price,
                                      win_rate=0.5, avg_win=0.02, avg_loss=0.01):
        profit_loss_ratio = avg_win / avg_loss if avg_loss > 0 else 1
        kelly_f = (win_rate * profit_loss_ratio - (1 - win_rate)) / profit_loss_ratio
        safe_kelly = max(0, kelly_f) * self.kelly_fraction  # ¼ Kelly
        position_value = account_value * safe_kelly
        return int(position_value / current_price)

# Factory for runtime selection
def create_position_sizer(strategy_name: str, config: dict) -> PositionSizer:
    strategies = {
        "fixed_percentage":  FixedPercentagePositionSizer,
        "kelly_criterion":   KellyCriterionPositionSizer,
        "fixed_shares":      FixedSharesPositionSizer,
        "volatility_adjusted": VolatilityAdjustedPositionSizer,
    }
    return strategies[strategy_name](**config)
Category 3 — Databases

Database Schema & Query Optimization

Artifact: DataStorage/dataStorage.py · DataStorage/schema_enhanced.py

View Code ↗

Artifact Description. The original schema was a single denormalized table with no indexes — adequate for small test datasets but unusable at scale. This enhancement made two major improvements: (1) four strategic B-tree indexes chosen based on the engine's actual query patterns, yielding a measured 50–100× speedup on a 25-million-row dataset; and (2) an enhanced normalized schema in schema_enhanced.py introducing a separate tickers master table, foreign key enforcement, CHECK constraints on all OHLCV values, an audit table for compliance, and data lineage tracking columns.

Justification for Inclusion. Database optimization is the discipline where you cannot guess — you must profile and measure. The four indexes were not chosen intuitively; they were derived by analyzing the WHERE clause patterns of every query the engine executes. Composite index column order is deliberate: (ticker, date) is ordered to support the engine's most common query shape (filter by ticker, then narrow by date range). The secondary composite index (date, ticker) supports cross-ticker analytics. This kind of query-pattern-driven index design is what separates a production database from a prototype.

Reflection. The most important learning here was that performance at 1,000 rows tells you nothing about performance at 100 million rows. Without indexes, PostgreSQL executes a full sequential scan regardless of selectivity. With the right composite index, the planner uses an index range scan that retrieves only the rows needed. The 50–100× figure is not a coincidence — it maps directly to the cardinality reduction from scanning ~1% of a 25M-row table versus all of it. Writing the schema comparison documentation (in schema_enhanced.py) forced me to articulate every design decision, which itself revealed two columns I had forgotten to include in the initial draft.

Lessons Learned

Key Reflections

The five most durable insights from the enhancement process.

⚡ Mathematical optimality ≠ practical optimality

Full Kelly Criterion is theoretically optimal in expectation but produces aggressive, high-variance positions. Professional traders use ¼ Kelly to reduce variance while preserving most of the return advantage. The math alone doesn't tell you this — it requires domain knowledge layered on top of algorithmic rigor.

🔧 Configuration is infrastructure, not an afterthought

Moving 47 parameters to YAML revealed that configuration has its own design axes: where parameters live, how validation happens, how environments coexist. Treating configuration as a first-class architectural concern — not a cleanup task — changes how you structure the entire system.

📊 You can't optimize what you don't measure

Index selection is not intuitive. I had to analyze every query's WHERE clause patterns before choosing composite index column order. Performance at 1,000 rows tells you nothing about performance at 25 million rows — the two scales require completely different reasoning.

✍️ Writing documentation improves the design

Every time I wrote a doc section explaining a design decision, I discovered a gap in the design itself. "Why does this index improve performance?" forced me to understand query plans. "When should you use Kelly vs. Fixed Percentage?" forced me to compare trade-offs I had only implicitly considered.

🔗 Integration is harder than components

Implementing each enhancement independently was straightforward. Making them work together coherently — configuration flowing into the position sizer, which flows into the engine, which writes to the indexed database — was the real design challenge. The glue code matters as much as the components.

🔒 Backward compatibility is a professional obligation

Every enhancement was designed so that existing code continued to work unchanged. Default parameters, optional arguments, and clear migration paths meant no breaking changes. In a team environment, breaking changes impose cost on every other developer — this must be treated as a professional obligation.
